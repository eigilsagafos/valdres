Creates a valdres instance — the Vue plugin plus a handle on its store —
mirroring the Pinia shape (`createPinia()` returns an instance). The store is
created in the `createValdres()` body, so `valdres.store` is available
immediately; `app.use(valdres)` provides it to every component.

```ts
interface Valdres {
    install(app: App): void
    store: Store
}
```

## Usage

```ts
import { createApp } from "vue"
import { createValdres } from "valdres-vue"
import App from "./App.vue"

const valdres = createValdres()
createApp(App).use(valdres).mount("#app")
```

## With a custom store

By default `createValdres()` creates `store({ batchUpdates: true })`. Pass your
own to share it (a warning is logged if it isn't batched):

```ts
import { store } from "valdres"
import { createValdres } from "valdres-vue"

const myStore = store({ batchUpdates: true })
app.use(createValdres({ store: myStore }))
```

## Initialize

Seed the store before any component reads it:

```ts
createValdres({
    initialize: txn => {
        txn.set(themeAtom, "dark")
    },
})
```

## SSR / Nuxt

Create **one instance per request** (a shared module-level store would leak
state across users). On the server, `dehydrate(valdres.store)` into the payload;
on the client, pass it back as `hydrate`:

```ts
// server
import { dehydrate } from "valdres"
const valdres = createValdres()
app.use(valdres)
// …render…
nuxtApp.payload.valdres = dehydrate(valdres.store)

// client
const valdres = createValdres({ hydrate: nuxtApp.payload.valdres })
```

`hydrate` runs core `hydrate` during creation (before any read), resolving atom
names through the global registry. Only `name`d atoms/families transfer; an
unknown name (its module not yet imported under code-splitting) is warn-skipped.
Atoms with a codec schema round-trip `BigInt`/`Date`/`Map`/`Set` over plain
JSON. When both `initialize` and `hydrate` are given, `initialize` runs first so
hydrated values win.
